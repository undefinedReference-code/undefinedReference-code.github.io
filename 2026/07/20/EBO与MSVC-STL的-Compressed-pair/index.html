背景：空类型也要占空间吗？

后面介绍的空基类有所不同：它不是独立对象，而是另一个对象内部的子对象。满足条件时，编译器可以让它与派生类的其他部分共享存储，这正是 EBO 能够成立的基础。更严格的对象模型规则放在文末附录中说明。

EBO 是什么？

用一句话概括：

EBO：若某个基类是空类，派生类可以不为这个基类子对象单独占内存；但若把同样的空类当作成员放进去，通常至少占 1 字节并可能带来对齐填充。

EBO 并不会让 sizeof(Empty) 变成 0。Empty 作为独立的完整对象时仍然至少占 1 字节；只有当它成为基类子对象时，编译器才可以让它不再占用额外存储。MSVC、GCC 和 Clang 都会在满足对象模型约束时应用这种优化。

标准库实现也会利用 EBO 或类似的布局技术压缩无状态删除器，因此在常见实现中，unique_ptr<int> 通常和一个裸指针一样大。不过，这只是常见实现结果，C++ 标准并不保证 sizeof(unique_ptr<T>) == sizeof(T*)。

下一节我们把这个问题放到更具体的场景里：指针 + 删除器 如何打包，以及 EBO 如何帮空删除器「省」下那一字节和对齐开销。

EBO与指针删除器

上一节用 WithMember / WithBase 说明了 EBO 的通用原理。这一节把它落到更贴近标准库的场景：管理指针时，如何把「句柄 + 删除器」打包进一个对象，又不让空删除器白占内存。

为什么需要「指针 + 删除器」？

堆上的指针常用 delete 释放，但资源并不总是 new 出来的：

资源 释放方式
new T delete
fopen 得到的 FILE* fclose
malloc 得到的内存 free

因此我们需要一个删除器（deleter）：在对象销毁时，按约定的方式释放指针。标准库里的 std::unique_ptr<T, Deleter> 就是这个模型——除了保存指针，还要保存「怎么释放」的逻辑。

朴素打包：两个成员

最直白的实现是组合：删除器和指针各作为一个成员。

这和上一节的 WithMember 是同一类问题：当 D 是空类（例如默认的 delete 逻辑）时，deleter 成员仍会占至少 1 字节，并因对齐把对象撑大。

三句话收束本节：

  1. 空删除器 + 组合 → 对象往往比裸指针大一倍（本 Demo 里 16 vs 8）。
  2. 空删除器 + 私有继承 → 对象可以压成和裸指针一样大；unique_ptr<int> 走的就是这条路。
  3. 有状态的删除器 → 无论组合还是继承，都要为 deleter 的状态留空间，大小不会变成 8。

WithEbo 只是教学用的最小模型。MSVC STL 里，unique_ptr 通过 _Compressed_pair<deleter, pointer> 把「是否 EBO」的判断封装起来——下一节看它在标准库里的落地方式。

noexcept

上一节两个构造函数末尾都有 noexcept(...)。它表示：括号中的常量表达式为 true 时，函数是 noexcept 的；为 false 时，函数可能抛出异常。

类型特征 对应的初始化代码 含义
is_nothrow_constructible<_Ty1, _Other1> _Myval1(forward(_Val1)) 用 _Val1 构造第一个元素，是否不抛异常

conjunction_v<A, B> 就是逻辑与：两个子对象构造都 nothrow，整个构造函数才标 noexcept。

forward 会根据实参是左值还是右值，选中拷贝构造或移动构造；trait 检查的正是 会被选中的那个构造函数 是否 nothrow。

noexcept 到底是什么意思？

可以把它理解成对编译器的承诺：

「我不会用 throw 把 C++ 异常传出这个函数；若真的抛了，别走 catch，直接 std::terminate()。」

它 只管 C++ 异常，不管其他「出错方式」：

管不管 例子
管 throw std::bad_alloc()、throw std::runtime_error(...)
不管 空指针解引用、数组越界（UB）、abort()、SIGSEGV（core dump）

所以：一个标了 noexcept 的函数照样可能因为 UB 而崩溃——noexcept 只约束 异常传播，不保证「绝对安全」。

_Compressed_pair 为什么使用条件 noexcept？

_Compressed_pair 的构造函数本身没有额外的业务逻辑，它只是构造内部的两个子对象。因此，它能否保证不抛出异常，取决于 _Ty1 和 _Ty2 的相应构造是否都不抛出异常。

当两个子对象的构造都满足 is_nothrow_constructible 时，条件表达式为 true，_Compressed_pair 的构造函数就是 noexcept 的；只要其中一个子对象的构造可能抛出异常，条件表达式就为 false，_Compressed_pair 也不再作出不抛异常的保证。

这里的条件 noexcept 不是运行时判断，也不表示条件为 false 时构造一定会抛出异常。一个可能抛出异常的构造函数，在某次调用中仍然可以正常完成。这个声明表达的只是：_Compressed_pair 能否在编译期承诺“异常不会从本次构造中传播出去”。

所以，条件 noexcept 的作用只是准确描述 _Compressed_pair 自身的构造语义。无条件写成 noexcept 会在子对象真的抛出异常时触发 std::terminate()；完全不写 noexcept，又会丢失两个子对象都保证不抛时本可提供的信息。至于上层代码是否查询或利用这个信息，是调用方自己的设计，并不是 _Compressed_pair 在这里服务于某个特定组件。

附录：对象地址规则与两类布局优化

正文使用了“独立对象”这种直观说法。更严格地说，在排除标准规定的特殊情况后，可以先记住下面这个结论：

生命周期重叠的、同类型的不同完整对象，必须具有不同的地址。

这句话包含三个需要分别理解的概念：

  1. 什么是“生命周期重叠”；
  2. 为什么强调“同类型”；
  3. 什么是“完整对象”。

理解这三个概念后，我们再来看 EBO 和 [[no_unique_address]] 为什么允许子对象共享存储，以及这种共享仍然受到哪些限制。

什么是“生命周期重叠”？

两个对象的生命周期重叠，表示它们在某一段时间内同时存在。此时程序必须能够区分那些不允许重叠存储的对象。

什么是“完整对象”？

**完整对象（complete object）**是指不是任何其他对象的子对象的对象。换句话说，它不是基类子对象、成员子对象或数组元素。可以先把它理解成“最外层、独立创建出来的对象”：

这里必须区分“完整对象”和“子对象”：完整对象通常需要保持独立的对象身份；基类和成员则存在于另一个对象内部，可能适用允许存储重叠的特殊规则。数组元素虽然也是子对象，但仍必须按数组布局规则依次排列，不能因为类型为空就全部放到同一地址。EBO 和 [[no_unique_address]] 优化的是特定的可能重叠子对象，而不是把两个普通的完整对象或数组元素压到同一地址。

EBO 为什么是例外？

基类子对象不是完整对象。C++ 将基类子对象归入 potentially-overlapping subobject（可能重叠的子对象），因此空基类可以不占用独立存储，并与派生类的其他部分共享地址：

这里 Empty 子对象仍然存在，只是它可能具有零大小并与 value 所在的存储重叠。编译器仍需遵守对象身份规则；例如，同一完整对象中的多个相同类型空基类子对象不能被随意压到同一地址。

[[no_unique_address]] 为什么也是例外？

普通成员默认需要自己的存储。C++20 的 [[no_unique_address]] 会把被标记的非静态数据成员也归入 potentially-overlapping subobject。对于本文讨论的空类型，这意味着空成员可以与对象中的其他部分共享地址：

policy 仍然是一个真实存在的成员子对象，只是不一定占用独立字节。该属性也不意味着编译器可以无条件合并任何成员：同类型子对象的地址约束、对齐要求和其他对象模型规则仍然有效，而且最终布局仍与具体实现有关。
